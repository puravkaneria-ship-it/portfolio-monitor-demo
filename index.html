7.4Going live (server-backed) — what changes

  1. Swap localStorage for a Postgres-backed API (one row per portco-period). The schema mirrors the template field names — no model change needed.
  2. Replace the SharePoint stub with the live M365 sync (server-side OAuth, refresh on a 6-hour cadence, write provenance back into each cell).
  3. Add row-level permissions: each portco user only sees / edits their own row. Fund-team users see all.
  4. Wrap PDF ingestion (board decks, audited statements) with a server-side table-extraction pipeline; route to the same template fields with a confidence score.
  5. Optional: feed numeric series into an LLM ("explain this portco's quarter in 3 sentences") to auto-generate the prose in Tab 3 modals.

How to use this tool
  1. Tabs 1–6 are the read-only analytics — show these in IC, partner meetings, or LP calls.
  2. Tab 7 (Data Lab) is where portcos and analysts add data. Three paths:
    • Drop a file (CSV/XLSX/JSON) using the template — fastest for monthly updates.
    • Manually add a portco with the form — fastest for first onboarding.
    • SharePoint sync — currently a stub; activates the moment Microsoft 365 is connected on a non-Yale account.
  3. localStorage means each user's browser keeps its own copy. Use Snapshot session in Tab 8 to share a JSON across the team until you deploy a server-backed version.
  4. Per-portco access (each company sees only their row) is a server-side feature — not enforceable in browser-only mode. Add this in step 3 of the deployment list (Tab 7.4).
  5. "Marginal benefit" (Tab 4.3) answers the partner question "what did we add vs market drift?" — tweak the sector CAGRs in Settings (Tab 8) to match your house view.
